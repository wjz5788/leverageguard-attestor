## 项目简介 / Project Overview

**爆仓保（LiqPass）** 为散户提供高杠杆交易的安全入口。
通过“保费—赔付”机制，让原本风险厌恶的用户也能参与衍生品交易，
从而扩大链上交易量、增加手续费收入，为 Base 生态带来新的用户群体。

**LiqPass** provides retail traders with an insured gateway to high-leverage trading.
By introducing a “premium-payout” mechanism, it allows risk-averse users to participate in derivative markets safely,
expanding on-chain trading volume, boosting transaction fees, and attracting new users to the Base ecosystem.

## 二、为什么用户愿意绑定交易所信息 / Why Users Connect Exchange Data

### 核心目的 / Core Purpose

我们要求用户绑定的唯一目的，是为了**验证爆仓事件的真实性**。
当用户上传爆仓订单号时，系统通过只读接口核查：

* 订单是否真实存在；
* 时间、标的、杠杆、结算价等是否匹配；
* 是否确实为爆仓（字段含 `LIQUIDATION` / `ADL`）；
* 该订单确属用户本人而非伪造。

The sole purpose of connecting a read-only API is to **verify the authenticity of a liquidation event**.
When a user submits an order ID, the system checks—via read-only API—whether the order exists,
matches the parameters (symbol, leverage, settlement), contains a liquidation flag,
and belongs to the submitting account.

---

### 安全与最小授权 / Security and Minimal Authorization

* 一个自然人用户通常会拥有多个交易所账户，但在使用爆仓保时，只需绑定其中**一家交易所**即可；
* 授权类型为**只读权限（Read-Only）**，无交易、提币、转账能力；
* 在实际操作中，我们**建议将主账户与高杠杆账户分离**，以保障资金安全；
* 举例说明：

  * 若用户主力使用 **OKX** 进行日常交易，则建议绑定 **Binance** 账户作为保障账户；
  * 若主力在 **Binance**，则建议绑定 **OKX**；
  * 若同时活跃于两家，可选择 **Gate.io** 或其他备用交易所。
* 用户可在所选交易所内开设多笔高杠杆仓位，无需绑定多家平台；
* 所有数据仅在用户申赔时使用，我们不会持续监听或后台拉取。

A single retail user may own multiple exchange accounts,
but only **one** needs to be connected when using LiqPass.
All connections are **read-only**, with no trading, withdrawal, or transfer rights.
We recommend **separating the main account from high-leverage accounts** for safety.
For example:

* If the main account trades on **OKX**, bind **Binance** as the insured account.
* If mainly on **Binance**, bind **OKX**.
* If active on both, use **Gate.io** or another small sub-account.
  Data are accessed only when verifying claims—never continuously.

> **简言之 / In short:** 爆仓保的绑定逻辑是“最小必要授权 + 账户隔离”，
> 既能完成爆仓验证，又最大限度保护用户资金安全。
> **LiqPass follows “minimal authorization + account isolation” to verify truth while protecting user assets.**

## 三、产品与理赔流程 / Product & Claim Workflow

### 当前产品 / Current Product

仅提供基于**本金与杠杆倍数**的定额爆仓保障。
用户在购买时输入本金与杠杆，系统计算保费与赔付额。

Currently, LiqPass offers a **fixed-payout protection** based on trading principal and leverage multiplier.
Premiums and payout limits are auto-calculated.

适用人群：希望尝试高杠杆但担心“一次清零”的散户交易者。
Target users: retail traders seeking higher leverage with protection against total liquidation.

---

### 购买流程 / Purchase Steps

1. 连接钱包（如 MetaMask）并登录 / Connect wallet (MetaMask)
2. 选择交易所并填写只读信息 / Choose exchange & input read-only keys
3. 输入本金与杠杆参数 / Enter principal & leverage
4. 系统计算保费与赔付额 / System calculates premium & payout
5. 用户支付后保单生效 / Policy activates after payment

---

### 理赔流程（被动机制） / Claim Process (Passive)

爆仓保为**被动理赔机制**，不会主动检测用户交易。
LiqPass adopts a **passive claim model**—no automatic monitoring of trades.

1. 用户发生爆仓后上传订单号或 JSON 文件；

2. 系统在本地核对订单详情与保单；

3. 信息匹配则发起赔付；

4. 若自动核对失败，用户可上传截图，由人工复核。

5. User uploads order ID or JSON after liquidation.

6. Local script verifies order details against policy.

7. If matched, payout is triggered.

8. If mismatch, user submits proof for manual review.

> 当前采用“本地离线核对 + 摘要上链”方式，
> 因云服务器位于美国，访问 OKX / Binance 接口不稳定。
> 仅将订单摘要生成 **Merkle Root** 上链以公开验证。
>
> LiqPass currently uses **offline local verification + Merkle-root attestation**,
> since U.S. nodes cannot stably access OKX/Binance APIs.
> Only hashed summaries (not raw data) are written on-chain.

## 四、数据与隐私 / Data & Privacy

* 用户数据前端加密传输，仅存密文；
* 不保留完整交易明细，只保留核对摘要；
* 理赔完成后将摘要生成 **Merkle Root** 上链；
* 用户可随时解绑并删除绑定信息；
* 链上记录不含任何可识别信息。

All data are encrypted client-side, stored only as ciphertext.
No full trading history is retained—only hashed summaries.
Post-claim, the summary is written on-chain as a **Merkle Root** for auditability.
Users can delete binding anytime; on-chain data are non-identifiable.

## 五、定价与风控说明 / Pricing & Risk Control

### 定价原则 / Pricing Principle

保费由系统基于**风险概率、赔付上限与运营成本**的综合模型计算，
确保资金池长期可持续、公平透明。

Premiums are derived from an internal model combining risk probability, payout ceiling, and operating costs—
ensuring sustainable, fair, and auditable reserves.

---

### 风控与反欺诈 / Risk & Anti-Fraud

* 设等待期与限赔，防止套利；

* 识别异常账户与重复申赔；

* 启用黑名单与人工风控开关。

* Waiting period and payout limits prevent instant arbitrage.

* Duplicate and anomalous accounts auto-flagged.

* Manual override and blacklist safeguard the pool.
